# Conversation Skill Draft

単一セッションの痕跡から、具体的な SKILL.md ドラフトを 1 本起こす。

---

## 位置付け

- **このスキル**: 1 セッションを深掘りして具体的なスキル draft を生成
- **session-skill-sketch**: 複数セッション横断で繰り返しパターンを見つけて作成計画を立てる

両方同時には使わない。先に sketch で候補を絞り、その後 draft で具体化する流れでも可。

---

## データソース

`${CLAUDE_PROJECT_DIR}/.claude/plugin-workspace/sessions/<session_id>/` 配下:

- `user-prompts.jsonl` — ユーザー発言の原文 (最重要)
- `approved-commands.jsonl` — 承認されたコマンド
- `session-log.md` — セッション要約 (補助情報)

**session_id の決定:**

1. ユーザーが明示的に session_id を指定した場合はそれを使う
2. 指定がない場合は mtime が最新のセッションディレクトリを対象にする
3. 「1 つ前のセッション」「最新のセッション」「さっきのセッション」などの表現も mtime 最新として扱う

---

## ワークフロー

### Step 1: セッション特定とデータ読み込み

対象セッションディレクトリを特定し、3 ファイルを読み込む:

```bash
SESSION_DIR=$(ls -dt .claude/plugin-workspace/sessions/*/ | head -1)  # mtime 最新
```

**欠損ハンドリング:**
- `session-log.md` が無い: user-prompts.jsonl と approved-commands.jsonl だけで続行、「要約が無いので精度が低い可能性」をユーザーに伝える
- `user-prompts.jsonl` が無い: 続行不可。スキル draft の中核はユーザー発言なので、無い場合は session-skill-sketch の利用を推奨して終了
- `approved-commands.jsonl` が無い: 続行、ただしスクリプト系スキルの判断は controlled

### Step 2: ユーザー発言の構造化

user-prompts.jsonl から以下を抽出する:

#### 2-1. 初期指示
- セッション最初の 1-3 発言
- これは「スキルの description にそのまま使える表現」を含む

#### 2-2. 訂正・方向転換 【最重要】
- 「いや」「違う」「そうじゃなくて」「やっぱり」「待って」「ちがう」を含む発言
- 直前の Claude のアクション or 提案を特定し、「何を訂正したか」をペアで記録
- **これらは全てスキル本文にルール or 反例として反映する**

#### 2-3. 追加要件
- 「あと」「さらに」「も」を含む発言
- 最初の指示から漏れた要件 = スキル化時に最初から織り込むべき機能

#### 2-4. 明示の前提共有
- 「このプロジェクトでは」「前回は」「いつもは」「うちの」を含む発言
- スキルの「前提条件」セクションに転記する候補

#### 2-5. 承認・完了マーカー
- 「OK」「いいね」「それで」「完璧」「ありがとう」
- 直前のアクションが「このスキルの成功例」として確定した瞬間

### Step 3: コマンド・ツール使用の構造化

approved-commands.jsonl から:

- 使用した tool の種類と頻度
- Bash コマンドは timestamp 順に並べて「作業の時系列」を再構築
- 連続して承認されたコマンド列 = ワークフローの骨格

### Step 4: スキルタイプの判定

Step 2 と Step 3 の結果から、作成するスキルのタイプを決める:

| タイプ | 判断基準 |
|---|---|
| instruction-only | 2-4 (前提共有) と 2-2 (訂正) が中心、コマンド実行は副次的 |
| with-scripts | Step 3 に定型的なコマンド連鎖 (3 ステップ以上) がある |
| with-hooks | セッション中に「毎回やってる」「自動でやってほしい」の発言がある |

### Step 5: SKILL.md ドラフトの生成

以下のテンプレートに沿って生成する:

```markdown
---
name: <kebab-case-name>
description: "<60-200字、いつ使うか>。<トリガーキーワード羅列>。"
---

# <Title>

<1-2文の概要。Step 2-1 の初期指示を参考にする>

---

## いつ使うか

<具体的なユースケース>

## ワークフロー

### Step 1: ...
### Step 2: ...

## 前提条件

<Step 2-4 で抽出した前提をそのまま転記>

## 注意点・例外

<Step 2-2 で抽出した訂正を「〜してはいけない」「〜の場合は〜する」形式で列挙>

## 成功例

<Step 2-5 で「OK」が出たアクションの要約>
```

**重要なルール:**

- 訂正された内容は必ず「注意点・例外」に含める (暗黙知の証拠なので消してはいけない)
- 前提共有は本文の冒頭ではなく「前提条件」セクションに集約する (再利用時に見失わないため)
- Claude 側が勝手に補完した箇所は Step 6 で明示する

### Step 6: 自己レポート (暗黙知対策)

ドラフト生成後、以下をユーザーに報告する:

- **裁量で補完した箇所**: セッションに直接の証拠がないが、ドラフト生成時に自分の判断で埋めた前提
- **曖昧だった箇所**: ユーザー発言やコマンドから判断できず、仮置きで書いた箇所
- **カバーされていない可能性**: このセッションには無かったが、本来スキルに含めるべきと思われる機能

**この自己レポートは empirical-prompt-tuning の「不明瞭点」「裁量補完」と同じ役割を果たす。**
これを見てユーザーが補足すれば、スキルの初稿の精度が上がる。

### Step 7: ユーザーとの対話

- ドラフトと自己レポートを提示
- ユーザーが「このスキルを作る」と決めたら `~/.claude/skills/<name>/SKILL.md` に配置するか確認
- skill-creator / skill-authoring-guide プラグインがあれば委譲

## 出力例

```
## ドラフト生成結果

**対象セッション**: abc123 (2025-01-20 14:30 〜 15:45)
**使用したデータ**:
- user-prompts.jsonl: 18 発言
- approved-commands.jsonl: 42 コマンド
- session-log.md: 有り

---

### 生成したスキル draft

(SKILL.md の全文をコードブロックで表示)

---

### 自己レポート

#### 裁量で補完した箇所
- 「テスト実行後に必ず lint する」ルールを入れたが、セッション中には 2 回しか出現しなかった。常にそうなのか確認してほしい。

#### 曖昧だった箇所
- ユーザー発言「前と同じ感じで」の「前」が何を指すか不明。スキル本文では「過去のセッションパターンに従う」と仮置きした。

#### カバーされていない可能性
- このセッションでは Git 操作が出現しなかったが、通常のワークフローなら含めるべきかもしれない。
```

## 前提条件

- `.claude/plugin-workspace/sessions/<session_id>/user-prompts.jsonl` が存在すること
- `jq` がインストールされていること
- (推奨) session-log.md が存在すること。無くても動くが精度が下がる
