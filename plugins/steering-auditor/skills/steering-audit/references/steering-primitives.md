# ステアリングprimitiveの決定表

Anthropic『Steering Claude Code: skills, hooks, rules, subagents and more』の枠組み。Claude Codeの振る舞いを操る手段（primitive）と、それぞれを**いつ使うべきか**の判断基準。監査ではこの表が「正しい乗り物を選んでいるか」の判定基準になる。

## 7つの primitive

| primitive | 読み込み | 性質 | 何を入れるか |
|-----------|---------|------|-------------|
| **CLAUDE.md** | セッション開始時にロードされ、全セッション常駐 | advisory（助言的） | ビルドコマンド、ディレクトリ構成、monorepo構造、コーディング規約、チーム規範 |
| **rules** | 条件・パスにスコープされて適用 | advisory（条件付き） | 特定ディレクトリ/ファイル種別にだけ効かせたい方針 |
| **skills** | name+descriptionのみ開始時ロード、本体は発火時に展開 | procedural（手順的・モデル判断あり） | デプロイ手順、リリースチェックリスト、レビュー工程などの**ワークフロー** |
| **subagents** | name+description+toolリストが開始時ロード、本体は自動発火しない | context isolation / 並列 | 検索・探索など、メインコンテキストを汚さず別文脈で回したい作業 |
| **hooks** | ライフサイクルイベントで必ず実行 | mandatory / deterministic（決定的・強制） | フォーマッタ、lint、セキュリティチェック、ロギングなど**必ず走らせる**もの |
| **output styles** | 応答スタイルを変更 | 表現の制御 | 回答の口調・構造・コミュニケーション様式 |
| **system prompt 追記** | `--append-system-prompt` | グローバル指示 | セッション全体に効かせる短い追加指示 |

加えて **slash commands** は、複数ステップのワークフローを**1キーストローク**で呼び出すための入力ショートカット（多くは skill / command を起動する）。

## 中核の判断軸

ブログの核心は「**指示の性質に primitive を合わせる**」こと。新しい指示を足すとき、または既存設定を監査するときは次を問う：

1. **必ず実行されなければならないか？（決定的か）**
   - YES → **hook**。モデルの判断に委ねず強制する（セキュリティ、フォーマット、lint、監査ログ）。
   - 「絶対に」「例外なく」「コミット前に必ず」のような語が CLAUDE.md/skill にあるなら、それは hook の候補。

2. **一般に従ってほしい助言的ガイダンスか？**
   - YES → **CLAUDE.md**（全体）または **rules**（特定パス/種別に限定）。
   - 規約・コマンド・構成説明はここ。

3. **手順的なワークフローか？（複数ステップ＋モデルの判断が要る）**
   - YES → **skill**。デプロイ、リリース、レビュー工程など。CLAUDE.md に長い手順書を埋めない（常時コンテキストを圧迫する）。

4. **別文脈・並列・コンテキスト隔離が要るか？**
   - YES → **subagent**。重い検索・探索、メインスレッドを汚したくない調査。要約だけ返す。

5. **ワークフローを1キーストロークで呼びたいか？**
   - YES → **slash command**（多くは skill を起動）。

6. **応答の口調・様式を変えたいか？**
   - YES → **output style**。

## 監査で頻出するミスマッチ（アンチパターン）

| 観測された状態 | 問題 | 推奨 |
|---------------|------|------|
| 「コミット前に必ずlintせよ」が CLAUDE.md にある | advisoryなので飛ばされうる。決定的でない | **hook**（PreToolUse/Stop等）に移す |
| advisoryな方針（口調・好み）が hook スクリプトに埋め込まれている | 過剰に強制的・脆い | **CLAUDE.md / rules** に移す |
| 30行のデプロイ手順が CLAUDE.md に直書き | 全セッションでコンテキストを圧迫、手順的 | **skill** に切り出す |
| 常時onの規約が skill になっている | 発火を待つ必要がなく、毎回読ませたい | **CLAUDE.md / rules** に移す |
| 重い検索ロジックが本体 skill に同梱 | メインコンテキストを汚す | **subagent** に隔離 |
| 同じ指示が CLAUDE.md・skill・rules に重複 | 出典が分散し矛盾の温床 | 単一の出典に**統合** |
| slash command が存在するが対応する skill/command 実体と不整合 | 壊れた・重複したショートカット | skill を正しく起動するよう修正 or 削除 |
| description が曖昧で skill が発火しない／誤発火する | トリガー精度が低い | description にトリガー語・use/not-use を明記 |

## 設計ベストプラクティス（全primitive共通の臭い）

- **大文字 ALWAYS/NEVER/MUST の連発**はイエローフラグ。理由（WHY）を説明する方がモデルの理解を促す。
- **肥大化**: 常時ロードされるもの（CLAUDE.md、skillのSKILL.md本体）はリーンに。詳細は参照ファイルへ（progressive disclosure）。
- **重複**: 同じ指示を複数の場所に書かない。
- **過適合**: 少数の例に特化した硬い指示ではなく、汎化する書き方。
- **自己完結性**: 必要情報をその場に置き、過剰な参照読み込みを強制しない。

## まず何から手をつけるか（ブログの推奨）

> skills から始める（作成が容易で即効性がある）。決定的な強制が要るところに hooks を足す。並列やコンテキスト隔離が効くところに subagents を使う。

監査の改善ロードマップもこの優先順で提示するとよい。
