# primitive別 評価ルーブリック

`primitive-evaluator` サブエージェントは、評価対象の primitive type に対応するセクションだけを読み、各チェック項目を評価する。各findingには `severity`・`evidence`（具体的な引用・行）・`recommendation` を付ける。**エビデンスのない指摘は出さない。**

severity の目安：
- **critical**: 壊れている／セキュリティ・データ損失リスク／確実に誤作動する（不正なフロントマター、存在しないスクリプト参照、危険なhook）
- **high**: ベストプラクティスに明確に反し、実害が出やすい（必須処理がadvisory、descriptionが機能不全、重大な肥大化）
- **medium**: 改善すべきだが致命的でない（WHY不足、軽い重複、命名）
- **low**: 望ましい改善（軽微な文言、任意の最適化）

---

## skill（SKILL.md）

1. **フロントマター妥当性** — `name`（kebab-case、ディレクトリ名と一致）と `description` が存在し有効か。
2. **descriptionのトリガー精度** — 具体的なトリガー語・「いつ使う／使わない」を含むか。曖昧だと未発火・誤発火する。近接スキルとの境界が明確か。
3. **手順的か（primitive適合）** — 内容は手順的ワークフローか。単なる常時の助言なら CLAUDE.md/rules 候補としてメモ。
4. **progressive disclosure** — SKILL.md本体がリーンで、詳細が references/ に分離され、読むタイミングが明記されているか。本体が長大なら肥大化。
5. **WHYの説明** — ALL-CAPS MUST/NEVER の連発でなく理由を説明しているか。
6. **自己完結性** — 過剰な参照読み込みや探索を強いていないか。頻出情報は本体にインライン化されているか。
7. **スクリプト/テンプレート参照の健全性** — 参照する `scripts/`・`templates/`・`agents/` が実在するか。`${CLAUDE_PLUGIN_ROOT}` 等のパスが正しいか。
8. **汎化** — 少数例への過適合でなく、異なるパターンに耐える書き方か。

---

## hook（hooks.json / settings.json の hooks）

1. **イベントの妥当性** — `PreToolUse`/`PostToolUse`/`UserPromptSubmit`/`SessionStart`/`Stop`/`PermissionRequest` 等が目的に合っているか。
2. **matcherの正確さ** — ツール名のmatcherが意図通りか（広すぎ／狭すぎ／タイポなし）。
3. **mandatory適合** — hookに入れるべき「必ず走らせる決定的処理」か。advisoryな方針が埋め込まれていないか（→CLAUDE.md候補）。モデル判断を要する作業をhookで無理にやっていないか（→skill候補）。
4. **スクリプトの実在と健全性** — `command` のスクリプトが実在し実行可能か。終了コード・標準出力の扱いが適切か。
5. **副作用と安全性** — 破壊的操作・ブロッキング・無限ループのリスク。冪等か。失敗時にセッションを不必要に止めないか。
6. **パフォーマンス** — 高頻度イベント（PostToolUse等）で重い処理を走らせていないか。
7. **パスの可搬性** — 絶対パス直書きでなく `${CLAUDE_PLUGIN_ROOT}` 等を使っているか。

---

## claude_md（CLAUDE.md / CLAUDE.local.md）

1. **advisory適合** — 内容は助言的ガイダンス（規約・コマンド・構成）か。
2. **手順書の混入** — 長いステップ・バイ・ステップのワークフローが直書きされていないか（→skill候補）。
3. **mandatory処理の混入** — 「必ず」「例外なく」の決定的要求が文章で書かれていないか（→hook候補）。
4. **肥大化** — 常時ロードされる前提で簡潔か。冗長・古い情報・無関係な内容がないか。
5. **WHY** — ALL-CAPS命令の羅列でなく理由を説明しているか。
6. **スコープ** — 全体に効かせるべき内容か。特定パスだけなら rules 候補。
7. **重複** — skill/rules と同じ指示を重複させていないか。

---

## subagent（agents/*.md）

1. **単一責任** — 役割が1つに明確に絞られているか。
2. **入出力契約** — 受け取るパラメータと返す成果物（多くは要約/JSON）が明示されているか。
3. **context isolation適合** — 別文脈・並列・検索隔離という subagent の利点に合った用途か。
4. **toolリストの最小性** — 与えるツールが目的に対して過不足ないか（過剰権限はリスク）。
5. **自己完結した指示** — 親の文脈に依存せず、単独で実行可能な指示になっているか。
6. **第三者性** — 作成意図を推測させず、与えられた入力だけで判断する設計か（評価系subagentの場合）。

---

## slash_command（.claude/commands/*.md）

1. **実体の存在** — 起動先の skill/処理が実在するか。壊れた参照でないか。
2. **多ステップ価値** — ショートカットにする価値のある複数ステップのワークフローか。
3. **引数の文書化** — 受け取る引数・使い方が明記されているか。
4. **重複/不整合** — 既存 skill と無駄に重複していないか、内容が矛盾していないか。
5. **命名** — 名前が機能を表し、衝突しないか。

---

## output_style（.claude/output-styles/*）

1. **スコープ適合** — 応答スタイルの制御に限定され、手順や強制処理を紛れ込ませていないか。
2. **明確さ** — 望むスタイルが具体的に記述されているか。
3. **他primitiveとの非干渉** — CLAUDE.mdやskillの指示と矛盾しないか。

---

## mcp（.mcp.json / settings の mcpServers）

1. **設定妥当性** — サーバー定義（command/args/env）が有効か。
2. **シークレット安全性** — トークン等が平文でコミットされていないか（critical）。
3. **必要性** — 使われていない／重複するサーバーがないか。
4. **権限スコープ** — 付与ツール/権限が必要範囲に絞られているか。

---

## 横断的に常に見る臭い（全primitive）

- ALL-CAPS の MUST/NEVER 連発（WHY不足のシグナル）
- 肥大化（常時ロード物が長大）
- 重複（同一指示が複数の場所に）
- 過適合（少数例に特化した硬直した指示）
- 壊れた参照（存在しないファイル/スクリプト/パス）
- 矛盾（primitive間で指示が食い違う）

これらは個別findingにするか、深刻なら Phase 3（fit-checker）に引き継ぐためのメモにする。
