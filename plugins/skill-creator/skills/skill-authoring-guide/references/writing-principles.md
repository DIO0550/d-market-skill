## 簡潔さの原則

### デフォルト前提：Claudeは既に賢い

情報の各項目に対して以下を自問する：
- 「Claudeにこの説明は本当に必要か？」
- 「Claudeが既に知っていると仮定できるか？」
- 「この段落はトークンコストに見合うか？」

### 良い例（約50トークン）

````markdown
## PDFテキスト抽出

pdfplumberでテキストを抽出する：

```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

### 悪い例（約150トークン）

```markdown
## PDFテキスト抽出

PDF（Portable Document Format）ファイルは一般的なファイル形式で、
テキスト、画像、その他のコンテンツを含みます。テキストを抽出するには
ライブラリが必要です。PDF処理には多くのライブラリがありますが、
pdfplumberは使いやすく多くのケースに対応するため推奨されます。
まずpipでインストールしてから以下のコードを使用します...
```

簡潔版はClaudeがPDFやライブラリについて知っていることを前提にしている。

## 自由度の設定

タスクの脆弱性と変動性に応じて指示の具体性を調整する。

### 高自由度（テキストベースの指示）

**使用場面：** 複数のアプローチが有効、判断がコンテキスト依存、ヒューリスティックが指針

```markdown
## コードレビュープロセス

1. コード構造と組織を分析する
2. 潜在的なバグやエッジケースをチェックする
3. 可読性と保守性の改善を提案する
4. プロジェクト規約への準拠を確認する
```

### 中自由度（パラメータ付きスクリプト/疑似コード）

**使用場面：** 推奨パターンが存在、ある程度の変動は許容、設定が動作に影響

````markdown
## レポート生成

以下のテンプレートを基に必要に応じてカスタマイズする：

```python
def generate_report(data, format="markdown", include_charts=True):
    # データ処理
    # 指定形式で出力
    # オプションでグラフを含める
```
````

### 低自由度（特定のスクリプト、パラメータ少数）

**使用場面：** 操作が脆弱でエラーが起きやすい、一貫性が不可欠、特定の順序が必須

````markdown
## データベースマイグレーション

以下のスクリプトをそのまま実行する：

```bash
python scripts/migrate.py --verify --backup
```

コマンドの変更やフラグの追加は行わないこと。
````

### アナロジー

Claudeを道を探索するロボットとして考える：
- **両側が崖の狭い橋** → 安全な道は1つ。具体的なガードレールと正確な指示を提供（低自由度）
- **障害物のない広い野原** → 多くの道が成功に導く。大まかな方向を示して最良のルートを任せる（高自由度）

## 記述パターン集

### テンプレートパターン

出力形式にテンプレートを提供する。厳密さのレベルを要件に合わせる。

**厳密な要件の場合：**

````markdown
## レポート構造

必ずこの構造を使用する：

```markdown
# [分析タイトル]

## エグゼクティブサマリー
[主要な発見の1段落概要]

## 主要な発見
- データに裏付けられた発見1
- データに裏付けられた発見2

## 推奨事項
1. 具体的な実行可能な推奨
2. 具体的な実行可能な推奨
```
````

**柔軟なガイダンスの場合：**

````markdown
## レポート構造

以下はデフォルトの形式。分析内容に応じて判断すること：

```markdown
# [分析タイトル]
## エグゼクティブサマリー
[概要]
## 主要な発見
[発見に基づいてセクションを調整]
## 推奨事項
[具体的なコンテキストに合わせる]
```
````

### 例示パターン

出力品質が例に依存するスキルには、入力/出力ペアを提供する：

````markdown
## コミットメッセージ形式

以下の例に従ってコミットメッセージを生成する：

**例1:**
Input: JWT認証の追加
Output:
```
feat(auth): JWT認証の実装

ログインエンドポイントとトークン検証ミドルウェアを追加
```

**例2:**
Input: レポートで日付が正しく表示されないバグの修正
Output:
```
fix(reports): タイムゾーン変換時の日付フォーマット修正

レポート生成全体でUTCタイムスタンプを統一使用
```
````

### 条件分岐パターン

判断ポイントでClaudeを導く：

```markdown
## ドキュメント修正ワークフロー

1. 修正タイプを判断する：
   **新規コンテンツ作成？** → 「作成ワークフロー」に従う
   **既存コンテンツ編集？** → 「編集ワークフロー」に従う

2. 作成ワークフロー：
   - docx-jsライブラリを使用
   - ドキュメントをゼロから構築
   - .docx形式にエクスポート

3. 編集ワークフロー：
   - 既存ドキュメントを展開
   - XMLを直接修正
   - 変更ごとにバリデーション
   - 完了時に再パッキング
```

## ワークフローとフィードバックループ

### 複雑なタスクにはワークフローを使用

複雑な操作は明確なステップに分解する。特に複雑なワークフローにはチェックリストを提供する：

````markdown
## 処理ワークフロー

以下のチェックリストをコピーして進捗を追跡する：

```
進捗:
- [ ] Step 1: 入力ファイル解析
- [ ] Step 2: データ検証
- [ ] Step 3: 変換処理
- [ ] Step 4: 出力検証
- [ ] Step 5: 最終出力
```
````

### フィードバックループの実装

**定番パターン：** バリデータ実行 → エラー修正 → 再実行

```markdown
## ドキュメント編集プロセス

1. `word/document.xml` を編集する
2. **即座にバリデーション**: `python scripts/validate.py unpacked_dir/`
3. バリデーション失敗の場合：
   - エラーメッセージを確認
   - XMLの問題を修正
   - バリデーションを再実行
4. **バリデーション通過後にのみ続行**
5. 再構築: `python scripts/pack.py unpacked_dir/ output.docx`
```

## コンテンツガイドライン

### 時間依存情報を避ける

```markdown
# 悪い例
「2025年8月以前はは旧APIを使用してください。」

# 良い例
## 現在の方法
v2 APIエンドポイントを使用: `api.example.com/v2/messages`

## 旧パターン（非推奨）
<details>
<summary>Legacy v1 API</summary>
v1 API: `api.example.com/v1/messages`（サポート終了）
</details>
```

### 用語の一貫性

1つの用語を選び、スキル全体で統一する：

| 良い（一貫） | 悪い（混在） |
|--------------|-------------|
| 常に「APIエンドポイント」 | 「APIエンドポイント」「URL」「APIルート」を混在 |
| 常に「フィールド」 | 「フィールド」「ボックス」「要素」を混在 |
| 常に「抽出」 | 「抽出」「取得」「引き出す」を混在 |

## アンチパターン

### Windowsパスを使わない

```
✓ 良い: scripts/helper.py, reference/guide.md
✗ 避ける: scripts\helper.py, reference\guide.md
```

### 選択肢を提示しすぎない

````markdown
# 悪い例
「pypdf、pdfplumber、PyMuPDF、pdf2image のいずれかを使用できます...」

# 良い例
「テキスト抽出にはpdfplumberを使用する：
```python
import pdfplumber
```
スキャンPDFでOCRが必要な場合はpdf2image + pytesseractを使用する。」
````

### 深いネストを避ける

```markdown
# 悪い例（深すぎる）
SKILL.md → advanced.md → details.md → 実際の情報

# 良い例（1階層）
SKILL.md → advanced.md（直接情報あり）
SKILL.md → reference.md（直接情報あり）
SKILL.md → examples.md（直接情報あり）
```

## 改善のための原則

### WHYを説明する

ALWAYSやNEVERの大文字連発や超厳格な構造は**イエローフラグ**。可能な限り理由を説明してフレームを変え、モデルがなぜその行動が重要かを理解できるようにする。

```markdown
# 悪い例
ALWAYS use pdfplumber. NEVER use pypdf.

# 良い例
テキスト抽出にはpdfplumberを使用する。pdfplumberはテーブルレイアウトを保持し、
座標ベースの抽出をサポートするため、フォーム処理に適している。
```

今日のLLMはtheory of mindが優れている。良いハーネスを与えれば、機械的な指示を超えて本当に成果を出せる。

### テストケースで繰り返される作業をスクリプト化する

テスト実行のトランスクリプトを読み、全テストケースのサブエージェントが独自に似たヘルパースクリプトを書いている場合、そのスクリプトをスキルにバンドルすべき。

例：3つのテストケース全てで `create_docx.py` や `build_chart.py` が書かれたら、一度書いて `scripts/` に入れ、スキルにそれを使うよう指示する。

### 汎化する

フィードバックから特定の例への過剰適合ではなく、広いカテゴリのユーザー意図とシチュエーションに汎化する。スキルは何百万回も使われる可能性がある。

### 定性フィードバックを定量メトリクスより優先する

execution_reportの `ambiguity_points`（不明瞭点）と `discretionary_fills`（裁量補完）がイテレーション改善の一次シグナル。実行時間やトークン数の削減は本文を短くするだけで達成できるが、曖昧さの解消こそがスキル品質の本質。メトリクスの改善に目を奪われて、定性的な問題を見落とさないこと。

### プロンプトをリーンに保つ

役に立っていない部分は削除する。トランスクリプトを読み（最終出力だけでなく）、スキルがモデルに非生産的な作業をさせていないか確認する。
