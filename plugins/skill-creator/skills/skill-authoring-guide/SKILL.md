# Skill Authoring Guide

Agent Skillの設計・作成・改善を体系的に支援するガイド。
Anthropic公式ドキュメントのベストプラクティスに基づく。

全体のフローは以下の通り：

- スキルの目的と大まかなアプローチを決める
- ドラフトを書く
- テストプロンプトを作成し、スキル付きClaudeで実行する
- ユーザーと一緒に結果を定性的・定量的に評価する
  - ラン実行中にアサーションを作成し、ユーザーに結果を見せる
- フィードバックに基づいてスキルを改善する
- 満足するまで繰り返す
- テストセットを拡大して再試行

ユーザーがこのプロセスのどこにいるかを判断し、適切な地点から支援する。既にドラフトがあれば評価・改善から。「evalはいらない、一緒にバイブスで」と言われたらそれでもOK。

---

## ユーザーとのコミュニケーション

このスキルはコーディング経験の幅広い層に使われる可能性がある。技術的なリテラシーのコンテキスト手がかりに注意して、コミュニケーションを適応する。

デフォルトの目安：
- 「evaluation」「benchmark」はボーダーラインだが概ねOK
- 「JSON」「assertion」はユーザーが知っていることを確認してから使う

疑わしい場合は簡単に用語を定義してから使う。

## Phase 1: 意図の把握

以下を確認してユーザーの意図を明確化する：

1. **このスキルで何ができるようになるか？** — Claudeに追加される具体的な能力
2. **いつトリガーすべきか？** — ユーザーが使う言い回し、コンテキスト
3. **期待するアウトプット形式は？** — ファイル形式、構造、品質基準
4. **テストケースは必要か？** — 客観的に検証可能な出力があるスキルはテスト推奨。主観的な出力（文体、デザイン）は不要なことが多い

会話の中で既にワークフローが確立されている場合（例：「これをスキルにして」）、使ったツール、手順、修正点をまず会話履歴から抽出する。エッジケース、入出力フォーマット、成功基準について積極的に質問する。

利用可能なMCPがあれば研究に活用する。

## Phase 2: 構造設計

### 基本構造

```
skill-name/
├── SKILL.md              # メイン指示（トリガー時に読み込み）
├── references/           # 参照資料（必要時に読み込み）
├── scripts/              # 実行スクリプト（コンテキスト非消費）
├── templates/            # テンプレートファイル
└── assets/               # アセット（画像、フォント等）
```

### 3段階の読み込みモデル（段階的開示）

| レベル | タイミング | コスト | 内容 |
|--------|----------|-------|------|
| **L1: メタデータ** | 常に | ~100トークン | `name`と`description` |
| **L2: 本文** | トリガー時 | 5k未満推奨 | SKILL.md本文 |
| **L3: リソース** | 必要時 | 実質無制限 | references/, scripts/, assets/ |

**設計原則：**
- SKILL.md本文は **100〜200行が目安**。超えたらオーケストレーターパターンで分解する
- スクリプトは**実行**されるためコンテキストを消費しない
- 参照ファイルへのリンクは **1階層のみ**
- 100行超の参照ファイルには目次を含める

### 分解の判断基準

SKILL.mdの各セクションについて問う：
- **「次に何をするか決めるために必要？」** → SKILL.mdに残す（フロー、分岐、ルーティング、共通制約）
- **「作業を実行するために必要？」** → references/に移す（詳細手順、仕様、スキーマ、具体例）
- **「独立したサブタスクとして実行できる？」** → agents/に移す（レビュー、検証、並列実行タスク）

オーケストレーターパターンの詳細と具体例は [references/structure-patterns.md](references/structure-patterns.md) を参照。

## Phase 3: SKILL.md執筆

### YAMLフロントマター（必須）

```yaml
---
name: your-skill-name
description: "何をするか＋いつ使うべきかの両方を含める。"
---
```

**nameの制約:** 最大64文字、小文字・数字・ハイフンのみ、XMLタグ禁止、予約語（anthropic/claude）禁止

**descriptionの制約:** 最大1024文字、空不可、XMLタグ禁止、三人称で記述

**descriptionの書き方のコツ：**
- Claudeは100以上のスキルからdescriptionのみで選択 — 十分な具体性が必要
- やや積極的（pushy）にする — Claudeはスキルを使わない方向にバイアスがある
- トリガーキーワードとコンテキストを豊富に含める
- トリガー条件情報は全てここに入れる（本文ではない）

### 本文の執筆原則

詳細は [references/writing-principles.md](references/writing-principles.md) を参照。主要原則：

1. **簡潔さが鍵** — Claudeが既に知っていることは書かない
2. **WHYを説明する** — ALWAYS/NEVERの連発より理由の説明が効果的。Theory of mindを活用
3. **命令形を使う** — 「〜してください」ではなく「〜する」
4. **自由度を適切に設定** — タスクの脆弱性に応じてlow/medium/highを選ぶ
5. **一貫した用語** — 同じ概念に異なる単語を混在させない
6. **時間依存情報を避ける**

### テンプレート集

[templates/](templates/) ディレクトリを参照：
- `templates/instruction-only.md` — コードなしの指示スキル用
- `templates/with-scripts.md` — スクリプト付きスキル用
- `templates/multi-domain.md` — 複数ドメイン対応スキル用

## Phase 4: リソース作成

### 参照ファイル（references/）

- 100行以上のファイルには**目次**を冒頭に追加
- 用途を明記してリンク: `**フォーム詳細**: [references/forms.md](references/forms.md) を参照`

### スクリプト（scripts/）

スクリプトのメリット：生成コードより信頼性が高い、コンテキスト非消費、実行が速い、一貫性保証

設計原則：
- エラーを適切に処理する（Claudeに丸投げしない）
- マジックナンバーを避け、定数の理由をコメントで説明
- 実行方法を SKILL.md に明記する

---

## Phase 5: テスト・評価

### テストケースの作成

スキルドラフト作成後、以下の構成でテストケースを作成。ユーザーに確認してから実行。テストケースは `evals/evals.json` に保存（スキーマは [references/schemas.md](references/schemas.md) 参照）。

**シナリオ構成（最低3つ）：**
- **median** 1個 — 典型的な使用ケース。このスキルが最も頻繁に受けるタスク
- **edge** 1-2個 — 境界条件、異常入力、エラーが起きやすい状況
- **hold-out** 1個（推奨） — チューニング中は実行しない。イテレーション3以降、または収束判定時に初めて投入し、過適合を検出する

**expectations の設計：**
- 各期待値に `critical` フラグを設定。criticalは**全パス必須** — 1つでもフェイルすれば改善が必要
- 良いアサーションは**識別的**: スキルが正しく動けばパスし、失敗すればフェイルする。どちらでもパスするアサーションは無価値

```json
{
  "skill_name": "example-skill",
  "evals": [
    {"id": 1, "prompt": "典型タスク", "category": "median", "holdout": false,
     "expectations": [{"text": "必須要件", "critical": true}]},
    {"id": 2, "prompt": "エッジケース", "category": "edge", "holdout": false,
     "expectations": [{"text": "エラーハンドリング", "critical": true}]},
    {"id": 3, "prompt": "hold-outタスク", "category": "median", "holdout": true,
     "expectations": [{"text": "基本要件", "critical": true}]}
  ]
}
```

### テスト実行ワークフロー（一連の流れで止めない）

結果は `<skill-name>-workspace/` にスキルディレクトリの兄弟として配置。iteration-N/eval-N/ で整理。

**Step 1: 全ランを同時に起動（スキルあり＋ベースライン）**

各テストケース（`holdout: true` を除く）に対して、同じターンでサブエージェントを起動:
- **スキルありラン**: スキルパス、タスク、入力ファイル、出力先を指定
- **ベースラインラン**: 新規スキルなら「スキルなし」、改善ならスナップショットした旧版
- **信頼性モード**: ベンチマーク/収束判定時は各構成3並列。クイックイテレーション中は1回でOK

**実行レポートの要件**: サブエージェントへのプロンプトに以下を含める — 「タスク完了後、`outputs/execution_report.json` を作成する。スキルの指示で**不明瞭だった箇所**（ambiguity_points）と、指示に記載がなく**独自判断した箇所**（discretionary_fills）を記録する」。このレポートがイテレーション改善の最も重要なシグナルとなる。スキーマは [references/schemas.md](references/schemas.md) 参照。

**重要**: 再評価には必ず新規サブエージェントを使用する。同じインスタンスの再利用は前回のコンテキストを保持するため偽の改善を生む。

各テストケースに `eval_metadata.json` を作成（記述的な名前を付ける）。

**Step 2: ラン実行中にアサーションを作成**

待機中に定量的アサーションをドラフトし、ユーザーに説明する。良いアサーションは客観的に検証可能で記述的な名前を持つ。主観的スキルには無理にアサーションを当てはめない。

**Step 3: ラン完了時にタイミングデータをキャプチャ**

サブエージェント完了通知から `total_tokens` と `duration_ms` を即座に `timing.json` に保存。これが唯一のキャプチャ機会。

**Step 4: 採点・集計・ビューアー起動**

1. **採点**: `agents/grader.md` を読んで各アサーションを評価。`grading.json` に保存。`text`, `passed`, `evidence` フィールド必須。graderは `execution_report.json` も読み、不明瞭点・裁量補完をgrading結果に含める
2. **集計**: ベンチマークデータを `benchmark.json` に集計（スキーマは [references/schemas.md](references/schemas.md) 参照）。収束判定データも含める
3. **分析**: `agents/analyzer.md` を参照してベンチマークデータのパターンを分析。不明瞭点のトレンドと収束状態を評価

**Step 5: フィードバックを読む**

ユーザーが完了したら `feedback.json` を読む。空のフィードバック = 問題なし。具体的な不満があるテストケースに改善を集中する。

## スキルの改善

### 改善の考え方

1. **フィードバックから汎化する** — スキルは数百万回使われる可能性がある。少数の例への過剰適合やMUSTの連発ではなく、異なるメタファーやパターンを試す
2. **プロンプトをリーンに保つ** — 役に立っていない部分は削除する。トランスクリプトを読み、スキルが非生産的な作業をさせていないか確認
3. **WHYを説明する** — 大文字のALWAYS/NEVERや超厳格な構造はイエローフラグ。理由を説明してモデルの理解を促すほうが効果的
4. **テストケース間の繰り返し作業を見つける** — 全テストケースで独自に似たスクリプトを書いている場合、それをスキルにバンドルすべきサイン
5. **不明瞭点と裁量補完が改善の最大シグナル** — パス率やメトリクスより、execution_reportの `ambiguity_points` と `discretionary_fills` を優先する。実行時間の短縮は本文削減で簡単にできるが、曖昧さの解消がスキル品質の本質

### 1イテレーション1テーマの原則

各イテレーションでは**1つのテーマのみ**を修正する。最もインパクトの大きい不明瞭点（または裁量補完）を1つ選び、最小限の変更で対処する。

理由：複数の変更を同時に行うと、パス率が上がっても下がっても原因を特定できない。1つずつ変えれば、各変更の効果を正確に測定できる。

例外：2つの問題が同じセクションにあり、同じ編集で解決できる場合は組み合わせてOK。

各イテレーションの記録を `iteration_log.json` に保存する（スキーマは [references/schemas.md](references/schemas.md) 参照）。

### tool_usesによる構造的健全性チェック

パス率100%でも `total_tool_calls` が15以上なら、スキルの自己完結性が低い。AIが参照ファイルの読み込みや探索的な操作を過剰に行っている。対策：
- 頻繁に参照される情報をSKILL.md本文にインライン化
- 参照ファイルの読みタイミングを明記（「Step 2でreferences/forms.mdを読む」）

### イテレーションループ

1. execution_reportの不明瞭点から最もインパクトの大きい1件を選ぶ
2. スキルに最小修正を適用（1テーマ）
3. 全テストケース（holdout除く）を新しい `iteration-<N+1>/` に再実行
4. 採点・分析し、`iteration_log.json` に記録
5. ユーザーのレビューを待つ
6. フィードバックを読み、次のテーマを選択

### 収束・発散・打ち切り基準

**収束（チューニング完了）:**
連続2回のイテレーションで以下を全て満たす → 完了
- 新規不明瞭点が0
- パス率向上 ≤ 3ポイント
- ステップ数 ±10%以内
- 実行時間 ±15%以内

**発散（構造的書き直しが必要）:**
3回以上のイテレーションで不明瞭点の数が減少しない → パッチを重ねても改善しない。問題のあるセクションを別の角度から書き直す。

**打ち切りの目安:**
- 50→80%: 効率帯。1-2イテレーションで大幅改善が見込める
- 80→90%: 2-3イテレーション要
- 90→100%: 収穫逓減。残りの失敗が重要かどうか判断してから追求する

ユーザーはいつでも「止める」「続ける」を判断できる。

### hold-outテストの導入

イテレーション3（または収束基準を初めて満たした時点）で、hold-outシナリオを実行する。hold-outのパス率がチューニングセットより **15ポイント以上** 低ければ過適合の兆候。直近のイテレーションで追加した制約が特定のテストケースに過度に最適化されていないか確認する。

## ブラインド比較（上級）

2つのスキルバージョンのより厳密な比較が必要な場合に使用。独立したエージェントにどちらがどのバージョンか知らせずに品質を判定させる。

詳細は `agents/comparator.md` と `agents/analyzer.md` を参照。

オプション機能であり、サブエージェントが必要。通常はヒューマンレビューループで十分。

## description自動最適化

スキル作成・改善後にdescriptionのトリガー精度を最適化する。

### Step 1: トリガー評価クエリを生成

20個の評価クエリ（should-trigger / should-not-trigger の混合）を作成。具体的でリアルなクエリにする（ファイルパス、個人コンテキスト、カジュアルな表現を含む）。

should-not-trigger のクエリは**ニアミス**が最も価値がある — キーワードは共有するが実際は別のことが必要なケース。

### Step 2: ユーザーにレビューしてもらう

評価クエリのリストをユーザーに提示し、追加・修正してもらう。

### Step 3: 最適化

各クエリに対してdescriptionがトリガーするかを確認し、トリガー精度が低い場合はdescriptionを調整する。60%/40%でtrain/test分割し、過学習を防ぐ。各descriptionを3回実行して信頼性のあるトリガー率を取得。

### Step 4: 結果を適用

`best_description` をスキルのフロントマターに更新。ユーザーに before/after とスコアを報告。

詳細は [references/description-guide.md](references/description-guide.md) を参照。

## Phase 6: パッケージング

### バリデーション

`agents/reviewer.md` を読んでサブエージェントを起動し、スキルの品質を自動レビューする。
レビュー結果の `review.json` を確認し、`fail` や `warn` の項目があれば対処する。

サブエージェントが使えない環境では、以下を手動で確認する：
- YAMLフロントマターの `name` と `description` が制約を満たしている
- SKILL.md本文が100〜200行の目安以内
- 参照ファイルへのリンクが正しい
- 不要なファイルが含まれていない

### パッケージ化

スキルディレクトリをzipアーカイブ（`.skill` ファイル）としてパッケージ化する。

## Claude.ai固有の指示

Claude.aiではサブエージェントがないため一部の手順が変わる：

- **テスト実行**: 並列実行不可。スキルのSKILL.mdを読んで自分でタスクを実行する。ベースラインランは省略
- **結果レビュー**: ブラウザが使えない場合はビューアーを省略し、会話内で結果を直接提示
- **ベンチマーク**: 定量ベンチマークは省略。定性フィードバックに集中
- **description最適化**: `claude -p` が必要なため省略
- **品質レビュー**: `agents/reviewer.md` は使用不可。チェックリストに基づいて手動で確認する
- **既存スキルの更新**: インストール済みパスは読み取り専用の場合あり。`/tmp/skill-name/` にコピーして編集

---

## Cowork固有の指示

- サブエージェントが使えるので主要ワークフローは全て動作する
- ブラウザ/ディスプレイがないため結果はファイルに出力してユーザーに共有する
- フィードバックは `feedback.json` として保存される
- description最適化はCoworkで問題なく動作するが、スキルが完成してユーザーが承認するまで待つ

## チェックリスト

以下の項目は `agents/reviewer.md` で自動レビュー可能（Phase 6参照）。

### コア品質
- [ ] descriptionが具体的でトリガーキーワードを含む
- [ ] descriptionに「何をするか」と「いつ使うか」の両方がある
- [ ] SKILL.md本文が100〜200行の目安以内
- [ ] WHYの説明がMUST/NEVERの連発より優先されている
- [ ] 時間依存情報がない
- [ ] 用語が一貫している
- [ ] 参照ファイルへのリンクが正しい
- [ ] 参照ファイルの連鎖がない（1階層ルール）
- [ ] 100行超の参照ファイルに目次がある

### テスト
- [ ] 3つ以上の評価シナリオを作成
- [ ] ユーザーにレビューしてもらった
- [ ] 使用予定の全モデルでテスト済み
