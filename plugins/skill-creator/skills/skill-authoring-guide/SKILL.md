# Skill Authoring Guide

Agent Skillの設計・作成・パッケージングを体系的に支援するガイド。
Anthropic公式ドキュメントのベストプラクティスに基づく。

全体のフローは以下の通り：

- スキルの目的と大まかなアプローチを決める
- 構造を設計する
- ドラフトを書く
- リソース（references/scripts/templates）を作成する
- パッケージングする
- **テスト・評価・改善が必要なら `skill-evaluator` スキルへ** — eval作成、ベンチマーク、イテレーション改善、収束判定、ブラインド比較、description最適化はそちらが担当する

ユーザーがこのプロセスのどこにいるかを判断し、適切な地点から支援する。既にドラフトがあれば構造レビューや執筆改善から。作ったスキルの品質を測りたい・回したいという話になったら skill-evaluator に引き継ぐ。

---

## ユーザーとのコミュニケーション

このスキルはコーディング経験の幅広い層に使われる可能性がある。技術的なリテラシーのコンテキスト手がかりに注意して、コミュニケーションを適応する。疑わしい場合は簡単に用語を定義してから使う。

---

## Phase 1: 意図の把握

以下を確認してユーザーの意図を明確化する：

1. **このスキルで何ができるようになるか？** — Claudeに追加される具体的な能力
2. **いつトリガーすべきか？** — ユーザーが使う言い回し、コンテキスト
3. **期待するアウトプット形式は？** — ファイル形式、構造、品質基準
4. **テストケースは必要か？** — 客観的に検証可能な出力があるスキルはテスト推奨。主観的な出力（文体、デザイン）は不要なことが多い。テスト・評価を行う場合は `skill-evaluator` を使う

会話の中で既にワークフローが確立されている場合（例：「これをスキルにして」）、使ったツール、手順、修正点をまず会話履歴から抽出する。エッジケース、入出力フォーマット、成功基準について積極的に質問する。

利用可能なMCPがあれば研究に活用する。

## Phase 2: 構造設計

### 基本構造

```
skill-name/
├── SKILL.md              # メイン指示（トリガー時に読み込み）
├── references/           # 参照資料（必要時に読み込み）
├── scripts/              # 実行スクリプト（コンテキスト非消費）
├── templates/            # テンプレートファイル
└── assets/               # アセット（画像、フォント等）
```

### 3段階の読み込みモデル（段階的開示）

| レベル | タイミング | コスト | 内容 |
|--------|----------|-------|------|
| **L1: メタデータ** | 常に | ~100トークン | `name`と`description` |
| **L2: 本文** | トリガー時 | 5k未満推奨 | SKILL.md本文 |
| **L3: リソース** | 必要時 | 実質無制限 | references/, scripts/, assets/ |

**設計原則：**
- SKILL.md本文は **100〜200行が目安**。超えたらオーケストレーターパターンで分解する
- スクリプトは**実行**されるためコンテキストを消費しない
- 参照ファイルへのリンクは **1階層のみ**
- 100行超の参照ファイルには目次を含める

### 分解の判断基準

SKILL.mdの各セクションについて問う：
- **「次に何をするか決めるために必要？」** → SKILL.mdに残す（フロー、分岐、ルーティング、共通制約）
- **「作業を実行するために必要？」** → references/に移す（詳細手順、仕様、スキーマ、具体例）
- **「独立したサブタスクとして実行できる？」** → agents/に移す（レビュー、検証、並列実行タスク）

オーケストレーターパターンの詳細と具体例は [references/structure-patterns.md](references/structure-patterns.md) を参照。

## Phase 3: SKILL.md執筆

### YAMLフロントマター（必須）

```yaml
---
name: your-skill-name
description: "何をするか＋いつ使うべきかの両方を含める。"
---
```

**nameの制約:** 最大64文字、小文字・数字・ハイフンのみ、XMLタグ禁止、予約語（anthropic/claude）禁止

**descriptionの制約:** 最大1024文字、空不可、XMLタグ禁止、三人称で記述

**descriptionの書き方のコツ：**
- Claudeは100以上のスキルからdescriptionのみで選択 — 十分な具体性が必要
- やや積極的（pushy）にする — Claudeはスキルを使わない方向にバイアスがある
- トリガーキーワードとコンテキストを豊富に含める
- トリガー条件情報は全てここに入れる（本文ではない）

> descriptionのトリガー精度を定量的に最適化したい場合は `skill-evaluator` スキルの「description自動最適化」を使う。

### 本文の執筆原則

詳細は [references/writing-principles.md](references/writing-principles.md) を参照。主要原則：

1. **簡潔さが鍵** — Claudeが既に知っていることは書かない
2. **WHYを説明する** — ALWAYS/NEVERの連発より理由の説明が効果的。Theory of mindを活用
3. **命令形を使う** — 「〜してください」ではなく「〜する」
4. **自由度を適切に設定** — タスクの脆弱性に応じてlow/medium/highを選ぶ
5. **一貫した用語** — 同じ概念に異なる単語を混在させない
6. **時間依存情報を避ける**

### テンプレート集

[templates/](templates/) ディレクトリを参照：
- `templates/instruction-only.md` — コードなしの指示スキル用
- `templates/with-scripts.md` — スクリプト付きスキル用
- `templates/multi-domain.md` — 複数ドメイン対応スキル用

## Phase 4: リソース作成

### 参照ファイル（references/）

- 100行以上のファイルには**目次**を冒頭に追加
- 用途を明記してリンク: `**フォーム詳細**: [references/forms.md](references/forms.md) を参照`

### スクリプト（scripts/）

スクリプトのメリット：生成コードより信頼性が高い、コンテキスト非消費、実行が速い、一貫性保証

設計原則：
- エラーを適切に処理する（Claudeに丸投げしない）
- マジックナンバーを避け、定数の理由をコメントで説明
- 実行方法を SKILL.md に明記する

---

## Phase 5: テスト・評価・改善（skill-evaluator へ）

スキルのドラフトができたら、品質を測る・回す・改善するフェーズに入る。これは **`skill-evaluator` スキルが担当する**：

- eval テストケースの作成（median / edge / hold-out）
- スキルあり／ベースラインの並列ラン実行と採点・ベンチマーク
- 不明瞭点・裁量補完を起点にした1テーマ1イテレーション改善
- 収束／発散／打ち切り判定、hold-out による過適合検出
- ブラインド比較、description自動最適化

「作ったスキルのベンチマークを測りたい」「evalを回したい」「精度を上げたい」となったら skill-evaluator に切り替える。「評価はいらない、バイブスで」と言われたらこのまま Phase 6 へ進んでよい。

---

## Phase 6: パッケージング

### バリデーション

`agents/reviewer.md` を読んでサブエージェントを起動し、スキルの品質を自動レビューする。
レビュー結果の `review.json` を確認し、`fail` や `warn` の項目があれば対処する。

サブエージェントが使えない環境では、以下を手動で確認する：
- YAMLフロントマターの `name` と `description` が制約を満たしている
- SKILL.md本文が100〜200行の目安以内
- 参照ファイルへのリンクが正しい
- 不要なファイルが含まれていない

### パッケージ化

スキルディレクトリをzipアーカイブ（`.skill` ファイル）としてパッケージ化する。

---

## 環境固有の指示

### Claude.ai
- **品質レビュー**: `agents/reviewer.md` は使用不可。下のチェックリストに基づいて手動で確認する
- **既存スキルの更新**: インストール済みパスは読み取り専用の場合あり。`/tmp/skill-name/` にコピーして編集

### Cowork
- サブエージェントが使えるのでパッケージング前の自動レビューは動作する
- ブラウザ/ディスプレイがないため結果はファイルに出力してユーザーに共有する

---

## チェックリスト

以下の項目は `agents/reviewer.md` で自動レビュー可能（Phase 6参照）。

### コア品質
- [ ] descriptionが具体的でトリガーキーワードを含む
- [ ] descriptionに「何をするか」と「いつ使うか」の両方がある
- [ ] SKILL.md本文が100〜200行の目安以内
- [ ] WHYの説明がMUST/NEVERの連発より優先されている
- [ ] 時間依存情報がない
- [ ] 用語が一貫している
- [ ] 参照ファイルへのリンクが正しい
- [ ] 参照ファイルの連鎖がない（1階層ルール）
- [ ] 100行超の参照ファイルに目次がある
- [ ] H1と導入文がnameやdescriptionの単純な繰り返しになっていない
- [ ] テンプレートプレースホルダー（[具体的な指示]等）が残っていない

### テスト（skill-evaluator で実施）
- [ ] 3つ以上の評価シナリオを作成
- [ ] ユーザーにレビューしてもらった
- [ ] 使用予定の全モデルでテスト済み
